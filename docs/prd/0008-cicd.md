# PRD 0008: CI/CD

## Status

Draft

## Purpose

Define when and how ReplayLab should add continuous integration and delivery workflows.

## Users

- Maintainers reviewing pull requests.
- Contributors validating changes before merge.
- Future release maintainers publishing packages or tools.

## Requirements

- Add CI before the project starts accepting meaningful implementation PRs beyond scaffold-level changes.
- Run restore, build, and tests on pull requests.
- Keep the first workflow simple and repository-local.
- Avoid release automation until package boundaries and versioning are clearer.
- Do not add deployment or Docker publishing until those product slices exist.

## Initial CI Scope

- Restore the solution.
- Build the solution.
- Run `dotnet test ReplayLab.sln`.
- Trigger on pull requests and pushes to `main`.

## Future CD Scope

- Package validation.
- NuGet publishing.
- CLI tool publishing.
- Docker image publishing, if Docker becomes part of the product.
- Release notes generation.

## Acceptance Criteria

- Pull requests show automated build and test status.
- CI uses the documented .NET target for the repository.
- CI does not require private secrets for normal validation.
- CD is not added until release requirements are documented.

## Out Of Scope

- Deployment.
- Docker publishing.
- NuGet publishing.
- Private adapter validation.
- Workflows requiring proprietary secrets or infrastructure.

## Assumptions

- CI should come before or with the first concrete parser implementation.
- CD should wait until packaging and release requirements are clearer.
- GitHub Actions is the likely CI platform because the repository is hosted on GitHub.

## Open Questions

- Should CI be added immediately in the scaffold PR or in the first parser PR?
- Should CI test only the latest SDK or a target framework matrix?
- What branch protection rules should be required before external contributions?
